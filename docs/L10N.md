# VoiceDrop iOS 本地化规范

开发语言是 zh-Hans：**中文字面量本身就是本地化 key**，英文翻译放各 target 的
`Localizable.xcstrings`。代码里正常写中文；只需保证"用户可见的字符串都走本地化查找"。

## 哪些不用动（SwiftUI 自动本地化，字面量参数类型是 LocalizedStringKey）

`Text("…")`、`Button("…")`、`Label("…", systemImage:)`、`Toggle("…")`、`Picker("…")`、
`Section("…")`、`TextField("…")`、`.navigationTitle("…")`、`.alert("…")`、
`confirmationDialog`、`ContentUnavailableView`、`Menu("…")` 等——字面量直接写在这些参数
位置的，编译器自动做本地化查找，保持原样。AppIntents 的 `LocalizedStringResource` 参数同理。

## 哪些要包 `String(localized:)`

用户可见、但以普通 String 形态存在的：

- 先存变量再显示：`let msg = "上传失败"` → `let msg = String(localized: "上传失败")`
- `Text(变量)` 上游的硬编码中文
- `Text(cond ? "甲" : "乙")`（三元表达式让参数变成 String，不再自动本地化）→
  `Text(cond ? String(localized: "甲") : String(localized: "乙"))`
- 本地通知 `UNMutableNotificationContent` 的 title/body
- UIKit 弹窗、分享面板的用户可见文字
- 展示用的错误/状态文案
- 插值照包：`String(localized: "已录 \(n) 条")`（自动生成 `%lld` 格式 key）

## 哪些绝对不动

- log / print / debug 字符串
- 发给服务器的内容（API payload、挖矿指令、语音改稿指令等——那是数据）
- UserDefaults key、标识符、文件名构造（RecordingName）
- 参与 `==` 比较或解析逻辑的字符串（本地化会破坏逻辑；如果它既参与比较又展示，
  不要动，报告出来人工处理）
- 服务端下发、本地只是兜底默认值的文案（如 UIConfigStore 里可被服务端覆盖的字段）——
  本地化了也会中英混排，不动，报告出来
- 注释

## 原则

拿不准就不动并报告。包装只改字符串表达式本身，不改任何逻辑、类型、结构。
